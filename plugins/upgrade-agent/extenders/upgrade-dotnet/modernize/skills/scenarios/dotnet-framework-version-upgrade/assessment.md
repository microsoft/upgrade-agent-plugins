## Goal

Analyze target projects to assess the impact of upgrading to .NET Framework 4.8.1 (net481).

## Entry Criteria

- Pre-initialization complete (handled by `scenario-initialization` skill)
- If git repo: on the correct working branch
- `initialize_scenario` already called — working folder exists at `.github/upgrades/{scenarioId}/`
- Target framework confirmed as net481 in `scenario-instructions.md`

## Exit Criteria

- `assessment.md` created in workflow folder
- **Guided mode**: User has reviewed and approved the assessment
- **Automatic mode**: Assessment summary surfaced to user; proceed to planning immediately

---

## Step 1: Run Assessment Tool

Call `generate_dotnet_upgrade_assessment` on the selected solution, project, folder, or project list. Scope the assessment to the user's selected projects when the request is not solution-wide.

For a solution-wide assessment:

```json
{
  "inputMode": "solution",
  "paths": "{absolute-path-to-solution}",
  "targetFramework": "net481"
}
```

For scoped project assessments, pass one or more absolute project paths as a semicolon-delimited string:

```json
{
  "inputMode": "projects",
  "paths": "{absolute-path-to-project-1};{absolute-path-to-project-2}",
  "targetFramework": "net481"
}
```

For folder assessments, pass the absolute folder path:

```json
{
  "inputMode": "folder",
  "paths": "{absolute-path-to-folder}",
  "targetFramework": "net481"
}
```

### Interpreting Results for Framework Upgrade

Use the assessment generated by `generate_dotnet_upgrade_assessment` as the source of truth. Do not suppress, rewrite, reclassify, or invent issues. For .NET Framework 4.8.1 upgrades, add only scenario-specific interpretation around the generated findings.

When summarizing the generated assessment, include any reported issues and use these areas as context for interpreting the upgrade impact:

| Area | Relevance | Action |
|------|-----------|--------|
| **NuGet package compatibility** | High | Check whether packages support net481. Packages already supporting compatible .NET Framework versions often require no change, but follow the generated findings. |
| **Deprecated APIs** | Usually low | Note any generated deprecation findings between the current source framework and net481. |
| **Security vulnerabilities** | Medium | Flag packages with known CVEs — upgrade is an opportunity to update. |
| **API breaking changes** | Usually low | Framework-to-framework retargeting to net481 is typically lower risk than moving to modern .NET, but follow any generated API findings. |
| **Project format** | Informational | Record which projects are legacy vs SDK-style (affects TFM edit approach). |
| **Other breaking issues** | Varies | Include any other generated breaking issues that do not fit the categories above. |

## Step 2: Create Assessment Document

The tool writes `assessment.md` in the workflow folder and also creates machine-readable assessment artifacts. Follow the generated content. If additional scenario framing is needed, append or summarize around the generated assessment without changing the reported issue list.

The assessment summary should include:

1. **Summary**: Number of projects, current framework versions, target (net481).
2. **Project inventory**: Table of projects with current TFM, project format (legacy/SDK-style), package management (packages.config/PackageReference).
3. **Package compatibility**: Any packages that may not support net481, following the generated findings.
4. **Retargeting needs**: `packages.config` `targetFramework` attributes, package reinstall/restore needs, or HintPath updates that may be required after the TFM change.
5. **Risks**: Summarize risks from the generated findings and note whether they appear blocking for a net481 retarget.
6. **Recommendation**: Proceed / proceed with caution / investigate.

### Scenario Summary Template

Use this structure only as a lightweight summary of the generated assessment, not as a replacement for the tool output.

```markdown
# Assessment: .NET Framework Version Upgrade to net481

## Summary
- **Projects**: {count} projects in solution
- **Current frameworks**: {list of current .NET Framework TFMs}
- **Target**: .NET Framework 4.8.1 (net481)

## Project Inventory
| Project | Current TFM | Project Format | Package Mgmt |
|---------|-------------|--------|--------------|
| {name} | net461 | Legacy | packages.config |
| {name} | net48 | SDK-style | PackageReference |

## Package Compatibility
{List any packages with compatibility concerns, or "No compatibility issues found."}

## Retargeting Needs
{List packages.config retargeting, package reinstall/restore, or HintPath work needed, or "No package retargeting needed."}

## Risks
{List risks or "Low risk — minor framework version bump."}

## Recommendation
{Proceed / Proceed with caution / Investigate further}
```

## Step 3: Present to User

- **Guided mode**: Open `assessment.md` and present a brief summary. Wait for user approval before proceeding to planning.
- **Automatic mode**: Show the summary and proceed to planning immediately.

## Edge Cases

### All projects already on net481
If every project is already on net481, inform the user: "All projects are already targeting .NET Framework 4.8.1. No upgrade is needed." Exit the scenario.

### Mixed modern and Framework projects
If the solution contains both modern .NET (net8.0+) and .NET Framework projects, only include the .NET Framework projects in the assessment. Modern projects are out of scope for this scenario.

### No .NET Framework projects found
If no .NET Framework projects are found, inform the user and suggest using `dotnet-version-upgrade` instead.
